## 🌿 Ветки и git-флоу

Работаем через Merge Request.

```text
main
  └── develop
        └── feature/<task-name>
```

Рекомендуемый процесс:

1. Заберите актуальный `develop`.
2. Создайте ветку `feature/<task-name>`.
3. Реализуйте задачу.
4. Проверьте сборку локально.
5. Запушьте ветку.
6. Откройте Merge Request в `develop`.
7. После проверки команда сливает MR.
8. Pipeline на `develop` автоматически деплоит frontend на стенд.

Прямые push в `main` и `develop` могут быть ограничены. Если нет прав на merge или push, обратитесь к наставнику.

## 🔁 CI/CD и деплой

Pipeline подключается из платформенного репозитория:

```yaml
include:
  - project: "hackathon-2026/platform"
    ref: "main"
    file: "workflows/frontend.yml"
```

В этом репозитории описываются только переменные, специфичные для frontend команды. Jobs вручную добавлять не нужно.

Pipeline выполняет:

1. проверку переменных;
2. сборку Vite-приложения внутри Docker (multi-stage build);
3. сборку Docker-образа с nginx;
4. push образа в registry;
5. деплой контейнера на командный стенд;
6. healthcheck после деплоя.

После успешного pipeline frontend доступен по адресу вида:

```text
https://team-N.hack.kam-dev.ru
```

Точный номер команды и адрес стенда сообщат наставники.

## ⚙️ Переменные сборки

Переменные задаются в `.gitlab-ci.yml` этого репозитория:

```yaml
ENABLE_FRONTEND: "true"
EXTRA_BUILD_ARGS: "VITE_API_URL=https://api.team-N.hack.kam-dev.ru"
```

| Переменная | Назначение |
|---|---|
| `ENABLE_FRONTEND` | Включает или отключает сборку и деплой frontend (`true` / `false`) |
| `EXTRA_BUILD_ARGS` | Build-time аргументы для Dockerfile в формате `KEY=VALUE KEY2=VALUE2` |

### Как работает `VITE_API_URL`

Vite **вшивает** значение переменной в JS-бандл на этапе сборки. В коде React она доступна как:

```js
const API_URL = import.meta.env.VITE_API_URL || "http://localhost:8080";
```

Это означает:

- значение фиксируется в момент `npm run build` внутри контейнера;
- изменить URL без пересборки образа нельзя;
- если переменная не передана в сборку, приложение будет обращаться к `localhost`.

> ⚠️ **Важно:** платформенный скрипт читает `EXTRA_BUILD_ARGS` — именно это имя переменной.
> Переменная `BUILD_ARGS` не поддерживается и будет проигнорирована.

## 🧩 Примеры настройки

### Frontend с backend на стенде

```yaml
ENABLE_FRONTEND: "true"
EXTRA_BUILD_ARGS: "VITE_API_URL=https://api.team-N.hack.kam-dev.ru"
```

### Временно отключить деплой frontend

```yaml
ENABLE_FRONTEND: "false"
```

Job сборки и деплоя будут пропущены. Job `verify` по-прежнему выполняется.

### Несколько build-аргументов

```yaml
EXTRA_BUILD_ARGS: "VITE_API_URL=https://api.team-N.hack.kam-dev.ru VITE_BOT_URL=https://bot.team-N.hack.kam-dev.ru"
```

Значения разделяются пробелом. Платформа сама подставит `--build-arg` для каждой пары.

## 🌐 Роутинг React Router

`nginx.conf` настроен для поддержки client-side routing. Все запросы, не соответствующие статическим файлам, перенаправляются на `index.html`:

```nginx
try_files $uri $uri/ /index.html;
```

Это позволяет React Router самостоятельно обрабатывать навигацию на клиенте. Менять `nginx.conf` не нужно — если только у вашего проекта нет специальных требований к прокси или заголовкам.

## 🧪 Локальная проверка

Перед push желательно проверить, что проект собирается локально.

### Запуск в режиме разработки

```bash
npm install
npm run dev
```

Приложение будет доступно на `http://localhost:5173`.

В режиме разработки `VITE_API_URL` можно задать через файл `.env.local` (он добавлен в `.gitignore` и не попадает в репозиторий):

```env
VITE_API_URL=http://localhost:8080
```

### Проверка production-сборки локально

```bash
npm run build
npm run preview
```

`npm run preview` поднимает локальный сервер для проверки собранного `dist/`.

### Проверка сборки Docker-образа локально

```bash
docker build \
  --build-arg VITE_API_URL=http://localhost:8080 \
  -t frontend-local \
  .

docker run -p 8080:8080 frontend-local
```

Откройте `http://localhost:8080` — должен открыться собранный React-интерфейс.

## 🧯 Если что-то не работает

### Стенд не обновился после push

1. Откройте **CI/CD → Pipelines**.
2. Проверьте, что pipeline на нужной ветке прошёл успешно.
3. Если pipeline упал, откройте job с ошибкой.
4. Исправьте проблему и запушьте изменения снова.

### Pipeline красный, но код выглядит корректно

Обратитесь к наставнику. Возможно, проблема связана с registry, runner, deploy-host или CI/CD переменными.

### Frontend открывается, но API не отвечает (ошибка `NetworkError`)

Вероятные причины:

- `VITE_API_URL` не передан в сборку или содержит опечатку — в браузере будет виден `http://localhost:8080`;
- backend ещё не задеплоен или упал;
- CORS: backend не разрешает запросы с адреса frontend.

Проверьте, какой URL вшит в бандл:

```bash
grep -o "api\.team-.*\.hack\.kam-dev\.ru" dist/assets/index-*.js
```

### `VITE_API_URL` не попал в образ

Убедитесь, что в `.gitlab-ci.yml` используется именно `EXTRA_BUILD_ARGS`, а не `BUILD_ARGS`:

```yaml
# ✅ Правильно
EXTRA_BUILD_ARGS: "VITE_API_URL=https://api.team-N.hack.kam-dev.ru"

# ❌ Неправильно — игнорируется платформой
BUILD_ARGS: "--build-arg VITE_API_URL=https://api.team-N.hack.kam-dev.ru"
```

### Нужно изменить URL backend

Измените значение `EXTRA_BUILD_ARGS` в `.gitlab-ci.yml` и запушьте. Pipeline пересоберёт образ с новым URL.
